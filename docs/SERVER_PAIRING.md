# notx Server Pairing — Design Document

> **Status**: Implemented — all five phases (S1–S5) are complete.
> See `docs/SERVER.md` for the operational reference and
> `tests/docker/server_pairing_test.go` for the smoke-test suite.

---

## 1. Purpose

Server pairing enables two or more notx server instances to trust each other for
data-replication, federation, or gateway scenarios. The goal is a system where:

- A **joining server** can register itself with an **authority server** over a
  dedicated gRPC service.
- Registration is guarded by a shared **pairing secret** (a short, human-readable
  token) that an administrator generates on the authority and distributes
  out-of-band (e.g., by copy-paste, secure message, or secrets manager).
- Once registered, the authority issues the joining server a **mTLS client
  certificate** signed by the authority's internal CA.
- All subsequent gRPC calls between the two servers use that certificate for
  mutual authentication — the pairing secret is never used again.
- Certificates have a finite TTL (30 days) and can be **hard-revoked** by an
  admin at any time, immediately blocking the revoked server at the TLS
  handshake layer.
- Certificate renewal is fully **automated** — a paired server renews by
  presenting its current valid cert over the mTLS channel, with no secret
  needed.

This mirrors how device pairing works today but is designed for **server-to-server
trust** rather than user-to-device trust.

## 2. Vocabulary

| Term                     | Meaning                                                                                                                        |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------ |
| **Authority server**     | The notx instance that owns the CA and issues certs.                                                                           |
| **Joining server**       | A second notx instance that wants to be trusted by the authority.                                                              |
| **Pairing secret**       | A short-lived, single-use password generated by the authority admin. Used exactly once during initial registration.            |
| **Server certificate**   | An X.509 TLS client certificate (EC P-256) issued to a joining server by the authority CA.                                     |
| **Authority CA**         | A self-signed EC P-256 CA key-pair held only by the authority server; never exported.                                          |
| **Server URN**           | A `urn:notx:srv:<id>` URN that uniquely identifies a server instance.                                                          |
| **ServerPairingService** | The new gRPC service that handles the pairing protocol, added to `notx.proto`.                                                 |
| **Bootstrap listener**   | A second gRPC listener (port `50052`) that accepts TLS without a client cert, used only for the initial `RegisterServer` call. |

---

## 3. Why gRPC-Only

Device registration exposes both an HTTP endpoint and a gRPC endpoint. Server
pairing is intentionally **gRPC-only** because:

1. **mTLS is first-class in gRPC** — the transport credentials layer already
   supports loading per-connection client certificates, making cert-rotation
   and revocation straightforward.
2. **Streaming** — future cert-rotation or real-time revocation pushes benefit
   from gRPC server-streaming without retrofitting SSE or WebSockets.
3. **No extra HTTP surface** — servers are infrastructure components; they do
   not need a human-facing JSON API for their pairing handshake.
4. **Existing pattern** — `DeviceService` is already gRPC-only for sensitive
   operations; server pairing follows the same principle.

The `ServerPairingService` is defined inside the **existing `notx.proto`** file.
A separate proto file will not be created now; the file can be split in a later
refactor when overall proto size warrants it.

## 4. Server URN

A new URN object type `srv` is introduced, following the existing pattern in
`core/urn.go`:

```
urn:notx:srv:<id>
```

Examples:

```
urn:notx:srv:01HZX3K8SERVERID123456
urn:notx:srv:01HZX3K8AUTHORITYSELF0   ← authority's own URN (self)
```

The joining server **generates its own server URN** at first run using the
`urn:notx:srv:` prefix. The generated URN is persisted in `instance.json` and
reused on every subsequent startup — it is the stable identity of that server
instance. This mirrors how devices generate their own URNs today.

Namespace is **not** part of the URN. It is separate metadata
(`"namespace": "acme"`) used for grouping and display only. The authority
enforces **server URN uniqueness** as the cluster invariant — not namespace
uniqueness. Two servers in different namespaces that somehow share a URN would
be rejected; two servers in the same namespace with distinct URNs are both
accepted.

## 5. Authority CA

The authority server maintains a lightweight internal CA. This CA is **not** the
same as the server's existing TLS certificate (which clients use to verify the
server's identity). It is a separate EC P-256 key-pair used solely to sign
server-pairing certificates.

### 5.1 CA Lifecycle

| Event                              | Action                                                                                   |
| ---------------------------------- | ---------------------------------------------------------------------------------------- |
| First startup with pairing enabled | Auto-generate CA key-pair; store on disk under `<data-dir>/ca/`                          |
| Subsequent startups                | Load from disk                                                                           |
| Admin rotation                     | Generate new CA; existing certs become untrusted immediately (hard revocation by design) |

### 5.2 CA Storage Layout

```
<data-dir>/
  ca/
    ca.key          ← EC P-256 private key (PEM, file-mode 0600)
    ca.crt          ← Self-signed CA certificate (PEM, file-mode 0644)
```

These files are **never served over the network**. The `ca.crt` is the only part
distributed — embedded in every issued certificate chain and available via the
unauthenticated `ServerPairingService.GetCACertificate` RPC.

## 6. Pairing Secret

### 6.1 Properties

- Generated by the authority admin via `notx server pairing add-secret` (CLI) or
  the Admin UI.
- **Cryptographically random**: 128-bit entropy, base32-encoded → ~26 printable
  characters (human-copyable).
- **Short-lived**: configurable TTL (default 15 minutes). After expiry the secret
  is refused even if unused.
- **Single-use**: consumed on first successful registration; subsequent calls with
  the same secret return `codes.Unauthenticated`.
- **Stored as a bcrypt hash** on the authority — the plaintext is never persisted.
- **Scoped optionally**: an admin can attach a label (e.g., `"datacenter-b"`) so
  audit logs are meaningful.

### 6.2 Secret Format

```
NTXP-ABCDE-FGHIJ-KLMNO-PQRST-UVWXY
└──┘ └──────────────────────────────┘
prefix     26 base32 chars (130 bits)
```

The `NTXP-` prefix ("notx pairing") makes secrets visually distinct and allows
secret-scanning tools (GitHub, GitLab, etc.) to detect accidental leaks.

## 7. Protocol Flow

```
Admin (authority)          Authority Server              Joining Server
      │                           │                             │
      │  notx server pairing      │                             │
      │  add-secret --label "dc-b"│                             │
      │──────────────────────────>│                             │
      │  secret: NTXP-ABCDE-...   │                             │
      │<──────────────────────────│                             │
      │                           │                             │
      │  (out-of-band: paste      │                             │
      │   secret into joining     │                             │
      │   server config)          │                             │
      │                           │                             │
      │                           │<── RegisterServer ──────────│ (port 50052, TLS only)
      │                           │    server_urn               │
      │                           │    csr (PKCS#10)            │
      │                           │    pairing_secret           │
      │                           │    server_name              │
      │                           │    endpoint                 │
      │                           │                             │
      │                           │  validate secret ✓          │
      │                           │  sign CSR → cert            │
      │                           │  store server record        │
      │                           │  mark secret used           │
      │                           │                             │
      │                           │── RegisterServerResponse ──>│
      │                           │    server_urn               │
      │                           │    certificate (PEM)        │
      │                           │    ca_certificate (PEM)     │
      │                           │    expires_at               │
      │                           │                             │
      │                           │                             │  write cert+key to disk
      │                           │                             │  configure gRPC client creds
      │                           │                             │
      │                           │<═══ subsequent gRPC calls ══│ (port 50051, mTLS)
      │                           │      (mTLS, client cert)    │
      │                           │                             │
      │                           │                    (≥7 days before expiry)
      │                           │<── RenewCertificate ────────│ (port 50051, mTLS — cert proves identity)
      │                           │    server_urn               │
      │                           │    new csr                  │
      │                           │                             │
      │                           │── RenewCertificateResponse─>│
      │                           │    new certificate (PEM)    │
      │                           │    new expires_at           │
      │                           │                             │
      │                           │                             │  atomically swap cert on disk
      │                           │                             │  reload gRPC client creds
```

### 7.1 Step-by-Step

1. **Admin generates a pairing secret** on the authority server. The secret is
   printed once and stored as a bcrypt hash.

2. **Admin distributes the secret** out-of-band to the operator of the joining
   server (e.g., via a secrets manager, encrypted message, or CI variable).

3. **Joining server generates a key-pair and CSR** (Certificate Signing Request)
   locally using EC P-256. The private key **never leaves the joining server**.

4. **Joining server calls `RegisterServer`** on the authority's bootstrap
   listener (port `50052`, TLS without client cert), passing:
   - Its self-assigned `urn:notx:srv:<id>` URN (generated at first run,
     persisted in `instance.json`)
   - The CSR (PKCS#10, DER-encoded)
   - The pairing secret (plaintext — channel is already TLS-encrypted)
   - A human-readable name and advertised gRPC endpoint

5. **Authority validates** the pairing secret (bcrypt compare), ensures it has
   not been used and has not expired.

6. **Authority signs the CSR** using its CA, producing an X.509 certificate with:
   - `CN = <server_urn>`
   - `SAN DNS = <advertised endpoint hostname>`
   - Validity = 30 days (configurable, default `720h`)
   - `KeyUsage: DigitalSignature, KeyEncipherment`
   - `ExtKeyUsage: ClientAuth` (used only as a gRPC _client_ cert)
   - Key algorithm: EC P-256

7. **Authority returns** the signed certificate and the CA certificate.

8. **Joining server writes** the certificate and private key to its configured
   cert directory and configures its gRPC client to present this cert on all
   future calls to the authority.

9. **All subsequent gRPC calls** from the joining server to the authority are
   made over mTLS on port `50051` — the authority verifies the client cert
   against its CA at the TLS handshake layer. No secret is needed again.

10. **Automatic renewal** — when the joining server detects its cert is within
    7 days of expiry, it calls `RenewCertificate` over the mTLS channel
    (port `50051`) using its current cert as the authenticator. A new CSR is
    generated (the key may be rotated or reused), the authority issues a fresh
    cert, and the joining server atomically swaps the cert on disk and reloads
    its gRPC credentials without restarting.

## 8. Dual Listener Architecture

Two separate gRPC listeners run on the authority when pairing is enabled:

| Listener      | Port    | TLS mode                    | Exposes                                                                                                     |
| ------------- | ------- | --------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **Primary**   | `50051` | mTLS (client cert required) | `NoteService`, `DeviceService`, `ProjectService`, `ServerPairingService` (all RPCs except `RegisterServer`) |
| **Bootstrap** | `50052` | TLS only (no client cert)   | `ServerPairingService.RegisterServer` + `ServerPairingService.GetCACertificate` only                        |

**Rationale for dual listener over single-port interceptor:**

- The bootstrap listener has **no `ClientAuth`** set at the TLS layer, so there
  is nothing to verify and no risk of accidentally serving an unauthenticated
  request on the primary port.
- Firewall / network-policy rules are simple: `50052` is open only to trusted
  admin networks; `50051` is the standard mTLS surface.
- After the initial handshake, operators can completely firewall `50052` if
  desired and no existing paired servers are affected.

The bootstrap port is configurable via `--pairing-port` (default `50052`).

## 9. Admin UI Integration

Server pairing management (listing servers, revoking, generating secrets) is
exposed through the **existing Admin service** (`internal/admin/admin.go`).

The Admin server acts as a **thin HTTP→gRPC proxy**:

1. The Admin server holds an internal gRPC client that connects to the local
   authority's primary port (`50051`) over mTLS, using the server's own
   admin device cert (or a dedicated admin service cert).
2. The Admin HTTP API exposes pairing management as standard HTTP endpoints under
   `/v1/servers/...`.
3. The React Admin UI calls those HTTP endpoints — it never speaks gRPC directly.
4. Because the Admin server only binds on `localhost` (or a restricted interface),
   these endpoints are only reachable by someone with access to the admin port,
   maintaining the same access-control model as today.

This means no new infrastructure is required — the gRPC client layer is reused,
and the Admin UI gets server pairing for free through its normal HTTP API pattern.

### New Admin HTTP endpoints

| Method   | Path                          | gRPC call proxied                                         |
| -------- | ----------------------------- | --------------------------------------------------------- |
| `POST`   | `/v1/servers/pairing/secrets` | — (generates secret, stores hash, returns plaintext once) |
| `GET`    | `/v1/servers`                 | `ServerPairingService.ListServers`                        |
| `DELETE` | `/v1/servers/:urn`            | `ServerPairingService.RevokeServer`                       |
| `GET`    | `/v1/servers/ca`              | `ServerPairingService.GetCACertificate`                   |

## 10. Proto Definition

`ServerPairingService` is added to the **existing `notx.proto`** file. A
dedicated proto file may be introduced in a future refactor when the file
warrants splitting.

```proto
// ─────────────────────────────────────────────────────────────────────────────
// ServerPairingService — server-to-server registration and cert lifecycle
// ─────────────────────────────────────────────────────────────────────────────

service ServerPairingService {
  // RegisterServer is called once by a joining server to obtain its mTLS
  // client certificate. Authenticated by pairing_secret, not a client cert.
  // Only available on the bootstrap listener (port 50052).
  rpc RegisterServer(RegisterServerRequest) returns (RegisterServerResponse);

  // RenewCertificate renews an expiring certificate. The caller must present
  // its current valid client cert over mTLS (port 50051). No secret required.
  rpc RenewCertificate(RenewCertificateRequest) returns (RenewCertificateResponse);

  // GetCACertificate returns the authority's CA certificate in PEM format.
  // Unauthenticated — the CA cert is public. Available on both listeners.
  rpc GetCACertificate(GetCACertificateRequest) returns (GetCACertificateResponse);

  // ListServers returns all registered peer servers. Admin-only (mTLS, port 50051).
  rpc ListServers(ListServersRequest) returns (ListServersResponse);

  // RevokeServer immediately and permanently invalidates a server's certificate.
  // The TLS handshake from the revoked server will be rejected on next connect.
  // Admin-only (mTLS, port 50051).
  rpc RevokeServer(RevokeServerRequest) returns (RevokeServerResponse);
}

// ── RegisterServer ────────────────────────────────────────────────────────────

message RegisterServerRequest {
  string server_urn        = 1;  // urn:notx:srv:<id>  — self-assigned by joining server at first run
  bytes  csr               = 2;  // PKCS#10 CSR, DER-encoded (EC P-256 key)
  string pairing_secret    = 3;  // plaintext NTXP-... secret; MUST NOT be logged
  string server_name       = 4;  // human label, e.g. "datacenter-b replica"
  string endpoint          = 5;  // gRPC endpoint reachable by authority, e.g. "grpc.dc-b.example.com:50051"
  string declared_namespace = 6; // informational grouping label, e.g. "acme"; NOT an identity check
}

// Note on declared_namespace: the authority records this field as metadata for
// grouping and display purposes. It does NOT enforce namespace uniqueness.
// The cluster invariant is server URN uniqueness — two registration attempts
// with the same urn:notx:srv:<id> are rejected regardless of namespace.
//
// Note on namespaceAwareSecretStore rewrite: earlier implementations rewrote
// the "notx:srv:" prefix to a tenant-scoped namespace prefix at registration
// time. That rewrite is obsolete under the current URN model — server URNs are
// globally prefixed urn:notx:srv:<id> and carry no namespace component.
// Removal of this rewrite is tracked in NAMESPACE_AND_PAIRING_DESIGN.md Phase N3.

message RegisterServerResponse {
  string server_urn     = 1;
  bytes  certificate    = 2;  // PEM-encoded signed X.509 client cert (EC P-256, 30-day TTL)
  bytes  ca_certificate = 3;  // PEM-encoded authority CA cert (trust anchor)
  google.protobuf.Timestamp expires_at    = 4;
  google.protobuf.Timestamp registered_at = 5;
}

// ── RenewCertificate ──────────────────────────────────────────────────────────

message RenewCertificateRequest {
  string server_urn = 1;  // urn:notx:srv:<id>
  bytes  csr        = 2;  // New CSR — key may be rotated or reused at caller's discretion
}

message RenewCertificateResponse {
  bytes  certificate = 1;  // PEM-encoded renewed cert (new 30-day TTL from now)
  google.protobuf.Timestamp expires_at = 2;
}

// ── GetCACertificate ──────────────────────────────────────────────────────────

message GetCACertificateRequest {}

message GetCACertificateResponse {
  bytes ca_certificate = 1;  // PEM-encoded authority CA cert
}

// ── ListServers ───────────────────────────────────────────────────────────────

message ListServersRequest {
  bool include_revoked = 1;
}

message ListServersResponse {
  repeated ServerInfo servers = 1;
}

message ServerInfo {
  string server_urn  = 1;  // urn:notx:srv:<id>
  string server_name = 2;
  string endpoint    = 3;
  bool   revoked     = 4;
  google.protobuf.Timestamp registered_at = 5;
  google.protobuf.Timestamp expires_at    = 6;
  google.protobuf.Timestamp last_seen_at  = 7;
}

// ── RevokeServer ──────────────────────────────────────────────────────────────

message RevokeServerRequest {
  string server_urn = 1;  // urn:notx:srv:<id>
}

message RevokeServerResponse {
  bool revoked = 1;
}
```

## 13. Hard Revocation

Revocation is **hard** — when `RevokeServer` is called, the authority:

1. Marks the server record `revoked = true` in the repository.
2. Adds the certificate's serial number to an **in-memory deny-set** that is
   rebuilt from the repository on every server startup.
3. Registers a custom `tls.Config.VerifyPeerCertificate` callback on the primary
   mTLS listener that checks the presented client certificate's serial number
   against the deny-set **at the TLS handshake layer** — before any RPC reaches
   a handler.
4. A revoked server's connection is **torn down during the handshake**, returning
   a TLS alert. It cannot issue any RPC, renew its cert, or receive any data.

This is true hard revocation enforced at the transport layer, not a soft
interceptor check. The deny-set is small (one entry per revoked server) so the
lookup is O(1) and adds no measurable latency to the handshake.

```
Revoked server                Authority (primary, 50051)
      │                               │
      │─── TLS ClientHello ──────────>│
      │<── VerifyPeerCertificate ──── │  ← serial in deny-set → reject
      │    (TLS alert: bad cert)      │
      │                               │
      ✗ connection refused            │
```

## 14. Automatic Certificate Renewal

The joining server runs a background goroutine that checks cert expiry on a
configurable interval (default: every 6 hours). When the remaining validity
drops below the **renewal threshold** (default: 7 days), it automatically
calls `RenewCertificate` over the existing mTLS channel.

### Renewal flow on the joining server

```
[background ticker — every 6h]
    │
    ├─ load server.crt from disk
    ├─ parse expiry
    │
    ├─ if time_until_expiry > 7d  → do nothing
    │
    └─ if time_until_expiry ≤ 7d:
           ├─ load or generate server.key (EC P-256)
           ├─ build new PKCS#10 CSR
           ├─ call RenewCertificate (mTLS, port 50051)
           ├─ write new cert to server.crt.tmp
           ├─ atomic rename server.crt.tmp → server.crt
           └─ signal gRPC client to reload credentials
```

The private key may be rotated on every renewal (recommended) or kept stable
(simpler for debugging). The joining server chooses — the authority only sees
the CSR public key.

### Configuration

| Flag / Config key               | Default         | Description                                         |
| ------------------------------- | --------------- | --------------------------------------------------- |
| `--peer-renewal-check-interval` | `6h`            | How often the joining server checks its cert expiry |
| `--peer-renewal-threshold`      | `168h` (7 days) | Remaining TTL at which renewal is triggered         |

## 16. Joining Server: Client-Side Behaviour

When a joining server is configured with `--peer-authority` and `--peer-secret`,
it performs the following sequence at startup:

1. **Check for an existing cert**: if `<cert-dir>/server.crt` exists, is valid,
   and has more than `RenewalThreshold` remaining → load it, skip pairing, start
   the renewal background goroutine, and connect to the authority normally.

2. **If no cert or cert is expired**:
   - Load or generate EC P-256 key-pair at `<cert-dir>/server.key`.
   - Generate a PKCS#10 CSR.
   - Dial the authority's bootstrap port (`50052`) with plain TLS. Verify the
     authority's server cert using the system CA pool, or a pinned CA cert at
     `<cert-dir>/ca.crt` if it already exists.
   - Call `RegisterServer`.
   - Write returned PEM cert to `<cert-dir>/server.crt`.
   - Write returned CA cert to `<cert-dir>/ca.crt`.
   - **Zero the pairing secret from memory**. Log a recommendation to remove
     `--peer-secret` from the configuration now that the cert is issued.

3. **Connect to the authority** on port `50051` over mTLS, presenting
   `server.crt` + `server.key`, trusting the authority via `ca.crt`.

4. **Start the renewal goroutine** that runs on `RenewalCheckInterval` and
   triggers `RenewCertificate` when TTL falls below `RenewalThreshold`.

### Client cert directory layout

```
<cert-dir>/
  server.key    ← EC P-256 private key (never sent to authority; file-mode 0600)
  server.crt    ← signed cert received from authority (refreshed on renewal)
  ca.crt        ← authority CA cert (trust anchor; updated if CA is rotated)
```

## 17. Security Considerations

| #   | Consideration                         | Mitigation                                                                                                                                                               |
| --- | ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1   | Pairing secret intercepted in transit | Sent over TLS on port `50052`. Single-use + 15 min TTL.                                                                                                                  |
| 2   | Pairing secret leaked via logs        | `pairing_secret` field **must never be logged**. The gRPC logging interceptor must redact this field by method name.                                                     |
| 3   | Private key exfiltration              | EC P-256 private key never leaves the joining server. Only the CSR (public material) is transmitted.                                                                     |
| 4   | CA private key compromise             | Stored at `0600`. If compromised, rotate the CA — all existing certs become untrusted immediately due to hard revocation design. Future option: HSM / KMS-backed CA key. |
| 5   | Rogue server with a stolen secret     | Single-use + bcrypt hash at rest + 15 min TTL. Even if stolen, registration window is at most 15 minutes and the cert is immediately revocable.                          |
| 6   | Replay of a valid cert                | Hard revocation via serial deny-set enforced at TLS handshake. 30-day TTL limits the blast radius of an undetected compromise.                                           |
| 7   | Bootstrap port abuse                  | Port `50052` exposes only `RegisterServer` + `GetCACertificate`. Should be firewalled to trusted networks. Can be closed entirely after all servers are paired.          |
| 8   | Renewal race condition                | Cert file is written atomically (`server.crt.tmp` → rename). gRPC client reloads credentials only after the rename succeeds.                                             |

## 18. Audit Logging

Every pairing-related event emits a structured `slog` line at `INFO` level.
The `pairing_secret` field is **never** included in any log entry.

| Event                                 | Key fields                                                                 |
| ------------------------------------- | -------------------------------------------------------------------------- |
| Secret generated                      | `event=pairing_secret_created label= id= expires_at=`                      |
| Secret consumed (success)             | `event=pairing_secret_consumed server_urn= label= remote_addr=`            |
| Secret rejected                       | `event=pairing_secret_rejected reason=(wrong\|expired\|used) remote_addr=` |
| Certificate issued                    | `event=server_cert_issued server_urn= expires_at=`                         |
| Certificate renewed                   | `event=server_cert_renewed server_urn= expires_at=`                        |
| Server revoked                        | `event=server_revoked server_urn= revoked_by=`                             |
| TLS handshake rejected (revoked cert) | `event=server_handshake_rejected server_urn= serial=`                      |
| Renewal triggered (joining side)      | `event=cert_renewal_triggered server_urn= days_remaining=`                 |
| Renewal succeeded (joining side)      | `event=cert_renewal_success server_urn= new_expires_at=`                   |
| Renewal failed (joining side)         | `event=cert_renewal_failed server_urn= error=`                             |

## 19. Data Model — Server Record Storage

Server records are stored as JSON files on disk, mirroring the device record
pattern:

```
<data-dir>/
  servers/
    srv_<id>.json              ← server record (one file per peer; named by server URN id segment)
  pairing_secrets/
    <id>.json                  ← secret record (bcrypt hash only; plaintext never stored)
```

The file is named using the `<id>` segment of the server URN (i.e. the part
after `urn:notx:srv:`). No namespace prefix is included in the filename because
namespace is not part of server identity.

### Server record JSON

```json
{
  "urn": "urn:notx:srv:01HZX3K8SERVERID123456",
  "name": "datacenter-b replica",
  "endpoint": "grpc.dc-b.example.com:50051",
  "cert_pem": "-----BEGIN CERTIFICATE-----\n...",
  "cert_serial": "3a:4b:5c:...",
  "revoked": false,
  "registered_at": "2025-01-15T10:00:00Z",
  "expires_at": "2025-02-14T10:00:00Z",
  "last_seen_at": "2025-01-15T10:00:00Z"
}
```

> `cert_serial` is stored separately from the PEM so the deny-set can be
> rebuilt efficiently at startup without parsing every certificate.

### Pairing secret record JSON

```json
{
  "id": "sec_01932c4f89ab",
  "label": "datacenter-b",
  "hash_bcrypt": "$2a$12$...",
  "expires_at": "2025-01-15T10:15:00Z",
  "used_at": null
}
```

## 21. What This Document Is NOT

- **It is not a replication protocol spec.** Pairing establishes mutual trust.
  What data flows over that trust channel is defined in a separate document.
- **It is not a user authentication spec.** Server-to-server identity is
  independent of end-user identity.
- **It is not an HA/clustering spec.** Pairing is a trust prerequisite, not a
  clustering or consensus protocol.
- **It is not final on replication scope.** Once pairing is implemented, a
  follow-on design document will define which data is replicated, in which
  direction, and under what conflict-resolution rules.
